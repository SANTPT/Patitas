# Historia de Usuario
## ID
[FE-22]
## Título
Dashboard Super Admin — Gestión de admins y permisos
## Descripción
**Como** Super Administrador de la plataforma  
**Se desea** poder crear, listar, editar permisos y suspender cuentas de Admin/Centro desde mi dashboard  
**Para** mantener el control total sobre quién puede gestionar contenido y perfiles de niños.
## Criterios de Aceptación
### Escenario 1: Listado de administradores
```gherkin
Dado que el Super Admin navega a "/dashboard/super-admin/admins"
Cuando la vista carga
Entonces debe ver una tabla con todos los admins: nombre, email, centro, permisos activos, fecha de creación y estado (activo/suspendido)
Y debe haber filtros por estado y buscador por nombre o email
```
### Escenario 2: Crear nuevo Admin
```gherkin
Dado que el Super Admin hace clic en "Nuevo Admin"
Cuando el formulario de creación se abre
Entonces debe poder introducir nombre, email, nombre del centro y seleccionar permisos con checkboxes
Y al guardar, el nuevo admin debe aparecer en la tabla y recibir un email de activación (simulado en mock)
```
### Escenario 3: Gestión de permisos
```gherkin
Dado que el Super Admin hace clic en "Editar permisos" de un admin existente
Cuando el panel de permisos se abre
Entonces debe ver checkboxes para: publicar_contenido, gestionar_perfiles, gestionar_tienda, ver_reportes
Y los cambios deben guardarse al hacer clic en "Guardar permisos"
```
### Escenario 4: Suspender/Reactivar un Admin
```gherkin
Dado que el Super Admin hace clic en "Suspender" en la fila de un admin
Cuando confirma la acción en un diálogo de confirmación
Entonces el admin pasa a estado "suspendido" en la tabla con badge rojo
Y sus acciones de gestión quedan bloqueadas (el admin no puede iniciar sesión)
Y el Super Admin puede reactivarlo con el botón "Reactivar"
```
## Notas
- Datos mock: array de admins en `localStorage` bajo `patitas_mock_admins`.
- Los permisos se almacenan como array en el objeto usuario: `permissions: ['publicar_contenido', 'gestionar_perfiles']`.
- La contraseña del nuevo admin se genera automáticamente (mock) y se "envía" por email (solo log en consola).
- En backend: `GET /api/admin/users`, `POST /api/admin/users`, `PATCH /api/admin/users/:id/permissions`, `PATCH /api/admin/users/:id/status`.
