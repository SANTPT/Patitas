# Historia de Usuario
## ID
[FE-15] *(actualizada)*
## Título
Página de Gestión de Pedidos — Cambios, Devoluciones y Envíos por producto
## Descripción
**Como** cliente que realizó una compra  
**Se desea** poder gestionar cambios, devoluciones o incidencias eligiendo directamente el producto afectado, sin necesidad de buscar códigos ni bajar al footer  
**Para** resolver cualquier problema de forma autónoma y guiada en menos de 2 minutos.
## Criterios de Aceptación
### Escenario 1: Usuario logueado — selector directo de producto
```gherkin
Dado que el usuario autenticado navega a "/devoluciones"
Cuando la página carga
Entonces ve directamente sus productos comprados agrupados por pedido, con imagen, nombre, fecha y estado
Y puede seleccionar el producto afectado con un clic sin introducir ningún código
Y solo aparecen productos de pedidos con estado "entregado" (las devoluciones solo aplican a pedidos recibidos)
Y si no tiene pedidos entregados, ve un estado vacío con aviso de los pedidos en curso
```
### Escenario 2: Usuario no logueado — búsqueda por código de pedido
```gherkin
Dado que el visitante sin sesión navega a "/devoluciones"
Cuando escribe su código de pedido (ej: PT-48291) en el campo de búsqueda
Entonces el sistema muestra los productos de ese pedido para que elija cuál quiere gestionar
Y si el pedido no está entregado, se muestra un aviso bloqueante con el estado actual
Y si el código no existe, se muestra un error descriptivo con link a contacto
```
### Escenario 3: Selección de motivo
```gherkin
Dado que el usuario (logueado o no) seleccionó un producto
Cuando avanza al paso 2
Entonces ve 6 tarjetas de motivo: dañado/defectuoso, producto incorrecto, cambio de opinión, cambio de variante, no llegó, otro
Y cada tarjeta muestra el tipo de resolución (devolución gratuita, cambio urgente, etc.)
Y al seleccionar una, avanza automáticamente al formulario
```
### Escenario 4: Formulario contextual
```gherkin
Dado que el usuario seleccionó un motivo
Cuando el formulario carga
Entonces muestra solo los campos relevantes para ese motivo (ej: fotos solo para "dañado")
Y el email de contacto se pre-rellena con el del usuario logueado
Y el botón de envío está deshabilitado hasta completar motivo específico y email
```
### Escenario 5: Confirmación con pasos siguientes
```gherkin
Dado que el usuario completó el formulario y envió la solicitud
Cuando se registra correctamente
Entonces ve una pantalla de confirmación con los pasos exactos según el tipo de gestión
Y un botón "Gestionar otro producto" para reiniciar el flujo
```
## Notas
- El flujo tiene 3 pasos: elegir producto → elegir motivo → formulario contextual → confirmación.
- Usuario logueado: `getProductsByUser(userId)` filtra solo pedidos `status === 'delivered'`.
- Usuario no logueado: busca por código y selecciona producto del pedido encontrado. Solo continúa si `status === 'delivered'`.
- Solicitud de devolución solo disponible para pedidos entregados en ambos flujos.
- En backend: `POST /api/requests` con `{ productId, orderId, type, subReason, description, photos, contactEmail }`.
