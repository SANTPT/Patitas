
# Historia de Usuario
## ID
[FE-13]
## Título
Store global de autenticación con Pinia
## Descripción
**Como** equipo de desarrollo  
**Se desea** centralizar el estado de autenticación del usuario en un store de Pinia  
**Para** que todos los componentes accedan al estado de sesión de forma reactiva y consistente.
## Criterios de Aceptación
### Escenario 1: Estado inicial del store
```gherkin
Dado que la aplicación carga por primera vez
Cuando Pinia inicializa el store de auth
Entonces debe verificar si existe un token válido en localStorage o cookie
Y si existe, debe intentar recuperar los datos del usuario con GET /api/auth/me
Y si falla (token expirado), debe limpiar el estado y redirigir a login
```
### Escenario 2: Login desde el store
```gherkin
Dado que el componente LoginPage llama a authStore.login(email, password)
Cuando la acción ejecuta POST /api/auth/login exitosamente
Entonces el store debe guardar el token y los datos del usuario (id, nombre, email, rol)
Y el getter isAuthenticated debe retornar true
Y el header debe actualizarse reactivamente mostrando el nombre del usuario
```
### Escenario 3: Logout
```gherkin
Dado que el usuario autenticado hace clic en "Cerrar sesión"
Cuando authStore.logout() se ejecuta
Entonces el token debe eliminarse del almacenamiento
Y el estado del store debe resetearse a sus valores iniciales
Y el usuario debe redirigirse a "/" con el header mostrando de nuevo "Iniciar sesión"
```
### Escenario 4: Getter de rol del usuario
```gherkin
Dado que el store tiene el usuario autenticado
Cuando un componente consulta authStore.isAdmin
Entonces debe retornar true si el rol del usuario es "admin" o "superadmin"
Y false en cualquier otro caso
Y los componentes de administración deben usar este getter para mostrar u ocultar opciones
```
## Notas
- Usar `defineStore` de Pinia con la Composition API (setup stores).
- El token JWT debe decodificarse en el cliente solo para leer los claims (no para validación — esa tarea es del backend).
- Implementar interceptor de Axios para adjuntar el token en cada petición automáticamente.
- Considerar refresh token para sesiones de larga duración.
